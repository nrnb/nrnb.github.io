Idea 2: Co-authorship Social network App

Description: Biological networks can be visualized and analyzed using Cytoscape. Often researchers want to go beyond the network of proteins or genes and also look at the inter-connectedness between colleagues and institutions. Who tends to publish together? What institutions are most collaborative? Are there inter-disciplinary connections in my institution?


Goal: Create a visual summary of how individuals are connected. An existing App currently queries PubMed directly or takes in a file with a set of publications (from scopus or incites) and builds a co-publication network where nodes are authors and edges the number of publications two authors share. Different sources have different depths of data, for instance PubMed, the most widely accessible of the three main sources, has citation counts but they are limited to only those articles cited in pubmed central and its institution data is limited to the first author. Given that PubMed is the most accessible source we want to expand its functionality to include computed institution data as well as better citation count approximations, and H-index calculation for individuals or subsets of papers.

This functionality can be applied to several data types outside academia, whenever relationships between people is of interest. An example is professional social networks (e.g. linked-in), where individuals are connected by who they have worked with in the past or the twitter-verse where individuals are connected when they each reference the same (#) hash tag. This project will start with support for co-publication networks and extend to other networks like twitter and linked-in as time permits. Additionally, we want to expand our co-authorship cytoscape App to include generic file formats to easily create the above networks.

Existing userguide - http://baderlab.org/UserguideSocialNetworkApp

Idea 3: User Interaction on Biomedical Linked Data

During the GSoC 2012, we developed SemScape, a plugin that allows the use of Cytoscape as a browser for Semantic Web (Linked Data) knowledge bases. We are currently in the process of updating libs and packaging the plugin for the Cytoscape App Store.

The idea for this year’s project is to go one step further and to allow the use of Cytoscape to visually define queries in SPARQL. The basic idea is that from a graph, a sub-graph can be defined where a few nodes or edges can be marked as variables (and eventually filters can be associated to them): this is equivalent to a (simple) SPARQL query. The user can then define queries in a simple and interactive way, without knowing the underlying syntax, that can be stored and applied on different data sources.

The project can then be extended in three directions, depending on the interest of the student:

  • allowing users to push such queries to a repositories and hence share them (the current plugin support updates of queries from remote repositories)

  • Implementing more complex queries (unions, negation, …)

  • Implementing heuristics for query-selection: in the current plugin, queries can be performed on demand, but they are also associated to specific actions. For instance, right-selecting a node of type protein can present as an option to expend the graph with the results of a query which is can include the specific protein as a value. This is possible through a simple markup through which queries are annotated by the user. Such markup could be included in the visual query/construction process and/or extracted from simple query patterns.


Some of the ideas in this project have been already prototyped, so we have a good understanding of the difficulty and timing required.

This project has the potential to make Cytoscape an ideal tool to navigate and query Linked Data resources (both in biomedical and, more generally, in Open Data domains). It also opens interesting possibilities in analyzing graphs from graph knowledge bases that can be represented as RDF (e.g.: dbpedia, freebase,...).


Language and Skills: Java/Maven. Knowledge of OSGi would be helpful. Knowledge of Semantic Web technologies would be a plus.

Idea 4: Cytoscape plugin for importing and visualizing variation data

Background: Variation data identifies genomic regions where individual base pairs are different from a reference genome. The consequences of these variations (e.g., stop gained in a protein sequence due to a SNP in a coding region) can influence gene regulation, structure and function, and are thus associated many genetic diseases. These data sets are large and difficult to navigate.


Goal: To extend and publish a Cytoscape 3 app in development that reads in variation data from various services and dynamically maps the data to biological networks and pathways independently (or selectively) loaded into Cytoscape. The app should provide the services needed to perform the mapping as well as default visualization settings, such as coloring all genes with variations in coding regions red.


The app currently supports:

  • Loading variation data from Ensembl REST services, VCF files, and GEMINI

  • Loading variation consequence data from Ensembl REST services, SnpEff-annotated VCF files, Ensembl VEP-annotated VCF files, and GEMINI

  • Predicting variation consequences with Ensembl VEP via Ensembl REST services

  • Summarizing variations and variation consequences per row/gene by Gene Ontology term

  • Visual mappings for variation and variation consequence counts


The app should be extended to provide:

  • Default color mappings for variation consequences by Sequence Ontology term

  • Advanced/compound visual mappings for counts of variation consequences summarized at the row/gene level (e.g., mutation diagrams, data rings, donut charts)

  • Variation service implementation based on ADAM

  • Variation service implementation based on a REST client for GEMINI

  • In-memory variation consequence prediction services based on SnpEff and JAnnovar

  • HGVS identifiers for variants where available (possibly via hgvs mapper)

  • Compare/contrast variation and/or variation consequences between samples (e.g. tumor vs normal) or populations (e.g. a case/control study)


Language and Skills: Java/Swing

Idea 5: CentiScaPe for multiple networks analysis and graphical output enhancement

Background: CentiScaPe is a Cytoscape plugin that allows to compute network centrality parameters. It is the most complete Cytoscape plugin for network centralities calculating Degree, Average Shortest Path, Eccentricity, Closeness, Betweenness, Centroid, Stress, Radiality, Eigenvector, Edge Betweenness, Bridging Centrality, for directed and undirected networks. CentiScaPe allows identifying network nodes that are relevant from both experimental and topological viewpoints. CentiScaPe also provides a Boolean logic- based tool that allows easy characterization of nodes whose topological relevance depends on more than one centrality. Finally, different graphic outputs and the included description of biological significance for each computed centrality facilitate the analysis by the end users not expert in graph theory, thus allowing easy node categorization and experimental prioritization.


Goal: The new beta version of CentiScaPe provides also the possibilities to analyze multiple networks and to integrate the analysis with experimental data set. This implies that a user can automatically start a computation of several networks and waits for the results. Anyway, the output of the analysis of a lot of networks (say 100) is useless if not supported by a smart and user friendly system to analyze the results. The Idea is to implements some techniques used in software analysis to extract information from the results of multiple computation in order to compare different networks and to extract topological based biomarkers:

  • The main framework for the results analysis is the Abstract Interpretation theory, particularly the interval analysis (and eventually others) [1].

  • Some statistical parameters, automatic analysis of the output, and new kinds of graphical outputs that can help to extract information from a multiple networks centrality analysis should be provided. The tools should for example be able to answer to questions as “How does the centroid value of the protein A chenge in the submitted 100 different networks?” or “How does the betweenness of protein B change if we consider 30 different phosporilation values for each proteins in the network N (this means we have 30 different experimental data set for the same network that must be included in the computation)?”.

  • The tools should also provide the implementation of the integration of experimental data-sets for the eigenvector, bridging centrality and edge betweenness (they are now available for all the other centralities).

Idea 7: Developing An Interface For The Cancer Network Altering Variant Database (CAN-VD) Using Cytoscape.js

Goals:

  1. Develop a web-based interface for PPI data retrieval with a comprehensive search function.

  2. Develop a visualization and annotation module using Cytoscape.js to visualize the retrieved PPI data as a network with annotations that highlight the mutation effects (the differences between the normal and cancer networks).

Idea 11: Enhance hierarchical models within Segway

Segway performs semi-automated annotation of the genome based on functional genomics data (ChIP-seq, RNA-seq, DNase-seq). Essentially it performs unsupervised clustering of these data anchored to the genome, using a dynamic Bayesian network framework. Since different patterns are visible at different scales, we could best describe genomic behavior using a hierarchical model, which would also allow us to better control additional parameters controlling transitions between segment labels and the emission of data. This would enable us to change the representation the relationship between these patterns from a simple network to a network of networks. Initial implementation of two-level hierarchical segmentation is included in Segway, but needs some improvement, such as:

  • add ability to do Viterbi decoding of both sublabels and superlabels separately

  • change superlabels so they emit sublabels instead of strict 1-to-many superlabel-sublabel transition

  • enable the use of hierarchical models in semi-supervised learning mode

  • add ability to enforce left-to-right movement through sequence

Idea 12: New Features for Interference App

Background: Interference is a Cytoscape plug-in for virtual knock-out experiments on complex networks. It allows to evaluate the topological effects of single or multiple nodes removal from a network. In this perspective, interference allows virtual node knock-out experiments: it is possible to remove one or more nodes from a network and analyze the consequences on network structure, by looking to the variations of the node centralities values. As the centrality value of a node is strictly dependent on the network structure and on the properties of other nodes in the network, the consequences of a node deletion are well captured by the variation on the centrality values of all the other nodes. With the interference app the user can:

  • Simulate pharmacological treatment: one can potentially predict side effects of the drug by looking at the topological properties of nodes in a drug-treated network, meaning with that a network in which a drug-targeted node (protein) was removed. To inhibit a protein (for instance a kinases) corresponds to removing the node from the network

  • Simulate gene deletion: gene deletion implies losing encoded proteins, thus resulting in the corresponding removal of one or more nodes from a protein network


Goal: The Goal of the proposal is to add more features to the App in order to consistently improving the tool:

  1. Calculation of centralities and other new node descriptors (node degrees, Markov Shannon entropies). These numbers will be used to rank the nodes for elimination; the user should choose the criteria (what node descriptor to be use for ranking, the default should be the node degree).

  2. Ranking the nodes using one node descriptor

  3. Eliminate 1 - x nodes and calculate each elimination step your entire network descriptors (ex: average node descriptors) and the Markov Shannon entropy (for the entire node) ; the user could choose the % or number of nodes to keep at the end of the network attack (ex: he can eliminate only 30% of the nodes using the node degree ranking)

  4. Presenting the results as text/table and plot.

  5. Interactive plot: the user can select one eliminated node into the final plot and see metabolic info, notations, semantics, etc. (the normal information you can have in Cytoscape)


More info at: http://www.cbmc.it/%7Escardonig/interference/Interference.php


Language and Skills: Java

Idea 22: A SPARQL query plugin for Pathvisio

Background: The semantic web is becoming a valuable resource for Pathway knowledge. The challenge still remain to smoothly integrate this knowledge into biological frameworks. With their SPARQL endpoints Uniprot (http://beta.sparql.uniprot.org/) or the EBI rdf Platform (https://www.ebi.ac.uk/rdf/) , biological data is becoming available as semantic web data. To use this knowledge one need not only to learn SPARQL queries, but also be prepared to convert the semantic data in different formats that comes with the various biological platforms. Using semantic web data in Pathvisio, we currently need to compose the appropriate SPARQL queries, get the data and write various parser to change the data in to the different data models.


Goal: To make a Pathvisio plugin that allows a pathway curator to write SPARQL queries against a variety of semantic web based pathways resources and include the obtained triples into the Pathway under scrutiny. The plugin should also provide an interface to SPARQL results for other Pathvisio plugins. Having such SPARQL interface still requires a pathway editor to become knowledgeable in writing SPARQL queries. The burden of having to write various parsers for semantic web data however, is taken away. The plugin should:

  • Provide a SPARQL query editor, from where the user could compose queries, which are then subsequently launched to the respective SPARQL endpoints.

  • Provide an interface that wrap specific SPARQL queries for other Pathvisio plugins.

  • Link to a centralized repository (e.g. Github) where SPARQL queries can be stored for future reference and others to use.

  • Link to a centralized index of relevant SPARQL endpoints. This index might again be stored on platforms like Github.

  • Import the resulting triples, which can then be linked to recognizable pathway components, or as data elements to use in Pathway analysis.

  • Make a RDF representation of a Pathway opened in Pathvisio. These triples are to be used as template for SPARQL queries. The user should be able to click on a triple which is then added to the WHERE block in the SPARQL query.


Potential students: This idea is for a student who has solid background in SPARQL queries, or the Semantic web in general, and who is looking into opening up the Semantic Web to specific knowledge domains (I.e. Biology and/or Medicine).

Idea 26: Porting Interference App to Cytoscape 3 with new features for undirected and directed networks

Background: Interference is a Cytoscape plug-in for virtual knock-out experiments on complex networks. It allows to evaluate the topological effects of single or multiple nodes removal from a network. In this perspective, interference allows virtual node knock-out experiments: it is possible to remove one or more nodes from a network and analyze the consequences on network structure, by looking to the variations of the node centralities values. As the centrality value of a node is strictly dependent on the network structure and on the properties of other nodes in the network, the consequences of a node deletion are well captured by the variation on the centrality values of all the other nodes. With the interference app the user can:

  • Simulate pharmacological treatment: one can potentially predict side effects of the drug by looking at the topological properties of nodes in a drug-treated network, meaning with that a network in which a drug-targeted node (protein) was removed. To inhibit a protein (for instance a kinases) corresponds to removing the node from the network

  • Simulate gene deletion: gene deletion implies losing encoded proteins, thus resulting in the corresponding removal of one or more nodes from a protein network


Goal: The Interference plugin is based on some features of the CentiScaPe plugin, and is available for Cytoscape 2.8.x but not yet for Cytoscape 3. The Idea is to port the plugin to Cytoscape 3 and to add some features. The plugin works for undirected networks, but now a new version of the CentiScaPe plugin allows to compute several centralities also for directed networks. The new tool should:

  • Work with Cytoscape 3.

  • Provide the Eigenvector, Bridging Centrality and Edge Betweenness

  • centrality Interference (now missing).

  • Provide the directed Interference for all the centralities implemented

  • More info at: http://www.cbmc.it/%7Escardonig/interference/Interference.php

Idea 28: New Features for Online Tutorial System

OpenTutorials is a free, online tutorial resource for open source and NRNB-supported tools. It is built on top of MediaWiki and a handful of extensions. It’s basic function is to provide a wiki-based collection of tutorials that can be viewed as a web page, slide show or printed handout. The wiki approach makes it easy to build and maintain content. The slide show and handout features make it useful for presenters.

There are a number of features that need to be implemented to improve the site and really make it effective to a larger community. What we’re looking for here with this project is an application that is very clear about the use cases of tutorials:

  • for developers wanting to create and maintain tutorial materials

  • for presenters of tutorials

  • and for users learning from tutorials.


The application should include a number of features that enhance these use cases. When possible, the features should point to other examples were similar approaches have proven successful (e.g., in other popular software). After we discuss the features, your application should then include implementation details and a timeline. These would be implemented as one or more PHP extensions, for example. Experience or some demonstration of familiarity with MediaWiki or PHP would be a plus. Here are some feature ideas to get you started:

  • Mix-and-match: We need a mechanism to allow presenters to mix and match tutorials (or sections of tutorials) into sets that are tailored for their class or workshop audience.

  • Transient locks: We need a mechanism for temporarily locking content while it is being presented (so that someone isn’t editing a wiki page while you are presenting it to a class of 100 students!). This can be done either by supporting temporary presenter “copies” that are not publicly editable, or simply locks that presenters can set for specific dates and times.

  • Enhanced image replacement tools to make it easier to update screenshots.

  • Tutorial builder tools using templates and forms to get a new tutorial started, or an integrated html-to-wiki converter like this.

  • Tweaks to the wiki page display of header/meta data and of slide breaks (e.g., a clear, yet unobtrusive icon).

Idea 36: Improved Layering/Interaction in Cytoscape 3 Networks

This project involves working on the visualization core in Cytoscape 3 (ding) to significantly improve the user experience. Currently, Cytoscape supports a single canvas to visualize the network. When moving or changing graphic objects (nodes and edges), the entire visible portion of the canvas needs to be updated. It has been pointed out by visualization experts that this problem can be easily solved by adding an additional “selection” canvas. Motion of nodes and edges could be performed in this canvas rather than the main canvas, significantly improving the interactivity on dense networks. The addition of a “selection” canvas also enables a number of visualization techniques that are too computationally expensive in the current Cytoscape model.
